Behind the Paper: Can LLMs Fix Schema Mismatches?

2026-03-30 3 min read
TL;DR

Schema drift in microservices is persistent toil. SAGAI-MID middleware uses an LLM to generate a translation function once, caches it, and runs it without the LLM after. The CODEGEN strategy achieves 0.90 pass@1, outperforming the naive DIRECT approach by 8 percentage points.

The Problem

In any microservice architecture that lives long enough, schema drift is inevitable. Service A sends a payload with a field called user_id. Service B expects userId. Service C was written six months later and expects account_identifier. The semantics are identical, but the schemas diverge because teams move at different speeds and coordination is expensive.

This is not a one-time problem. Every API version bump, every new service, every refactor creates the potential for new mismatches. The industry response has been schema registries, contract testing, and API gateways, all of which help but none of which eliminate the problem. Teams still spend significant engineering hours writing and maintaining bespoke translation layers between services that should, in principle, understand each other.

Our Approach

SAGAI-MID is a middleware layer that sits between microservices and resolves schema mismatches using large language models. We evaluated two strategies. The DIRECT strategy feeds the source payload and target schema to the LLM and asks it to produce the translated payload directly, on every request. The CODEGEN strategy instead asks the LLM to generate a translation function, which is then compiled, tested against sample payloads, and cached.

The distinction matters enormously in production. DIRECT is simple but puts an LLM call in the critical path of every request. Latency is unpredictable, cost scales linearly with traffic, and you depend on the LLM being available for every API call. CODEGEN pays the LLM cost once, at schema-discovery time, and from that point forward the cached translation function runs as ordinary code with deterministic latency and zero LLM dependency.

Results

On our benchmark of 120 schema mismatch scenarios drawn from real-world microservice architectures, CODEGEN achieved 0.90 pass@1, meaning 90% of generated translation functions passed all test cases on the first attempt. DIRECT achieved 0.82 pass@1. The 8 percentage point gap is explained by the nature of the task: when the LLM generates code, it can use typed function signatures and explicit field mappings, which are inherently more precise than freeform JSON-to-JSON translation.

Latency tells a sharper story. DIRECT adds 800-2000ms per request depending on payload complexity and LLM load. CODEGEN's cached functions execute in under 1ms. After the initial generation pass, the middleware behaves like a conventional, hand-written translation layer, just one that was written by an LLM instead of a human.

Key Insight

The important idea here is not that LLMs can translate JSON. It is the separation between generation time and execution time. The LLM is a compiler, not a runtime dependency. It writes the translation function once, the function is validated, and then the LLM is out of the loop. This pattern applies far beyond schema translation. Any time you are tempted to put an LLM call in a hot path, ask whether the LLM could instead generate a deterministic artifact that runs without it. The answer is yes more often than you might expect.

For teams dealing with schema drift at scale, SAGAI-MID reduces the maintenance burden from continuous manual translation work to a one-time review of auto-generated functions. The functions are readable code, not opaque model outputs, so engineers can inspect, modify, and version them like any other code artifact.
