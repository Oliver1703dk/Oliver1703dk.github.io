Behind the Paper: Flood Detection on Edge Devices

2026-03-30 3 min read
TL;DR

Architectural decisions matter more than model compression. An FSM-guided multi-model pipeline on cheap edge hardware can outperform a single large model in reliability, latency, and false-positive rate for agricultural flood detection.

The Problem

Flooding is one of the most destructive natural hazards for agriculture. Crops are wiped out in hours, and the economic damage compounds when early warning systems fail. The obvious solution is to deploy ML-based detection, but most farms sit in regions with intermittent or nonexistent connectivity. Cloud inference is not an option when the upstream link is a 2G modem that drops every afternoon. Any detection system that depends on a round trip to a data center is, by construction, unreliable in exactly the conditions where reliability matters most.

What We Did

Instead of taking a large model and compressing it until it fits on a Raspberry Pi, we reframed the problem as an architectural decision. We designed a finite-state machine (FSM) that governs which model tier runs at any given time. In normal conditions, a lightweight anomaly detector monitors sensor readings at minimal power cost. When the FSM detects a transition into an elevated-risk state, it promotes inference to a mid-tier model on the same Raspberry Pi, or to a Jetson Nano if available on the local network.

We also incorporated diurnal sensor fusion. Water-level sensors, soil moisture probes, and rainfall gauges have different noise profiles depending on time of day. The FSM encodes these patterns explicitly, adjusting detection thresholds and model selection based on the current diurnal phase rather than treating all readings uniformly.

What We Found

The multi-model consensus approach significantly reduced false positives compared to a single-model baseline. When two independent model tiers agree on an alert, the probability of a spurious trigger drops substantially. The FSM also bounds worst-case latency: because state transitions are deterministic and the model-selection logic is precomputed, there is no runtime search or negotiation involved. Alert latency stays within a predictable envelope regardless of sensor noise.

On hardware cost, the full pipeline runs on a Raspberry Pi 4 plus an optional Jetson Nano, total BOM under $200. Power consumption stays within the budget of a small solar panel, which matters for remote agricultural deployments.

Why It Matters

The core insight is not about flood detection specifically. It is about how you allocate inference resources on constrained hardware. The standard approach in edge ML is to take the best model you can find and compress it. That works, but it treats the deployment environment as a fixed constraint to work around. The FSM-guided tiering approach instead treats the environment as a signal: the deployment conditions themselves inform which model should be active. This pattern generalizes to any domain where conditions change over time and different risk levels justify different computational budgets.
